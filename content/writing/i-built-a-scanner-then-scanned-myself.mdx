A few weeks ago I shipped [lab.marwandiallo.com](https://lab.marwandiallo.com), a set of four hands-on security playgrounds I built on weekends to teach the failure modes I see most often in client work: identity (passkeys + JWT), Content-Security-Policy, prompt injection, and SSRF.

The labs are deterministic rule engines, not AI. They use the same pattern as Google's [csp-evaluator](https://csp-evaluator.withgoogle.com), Mozilla Observatory, and the passive scan rules in OWASP ZAP: parse the input, walk a small ruleset, return findings with severity and remediation.

I built them, shipped them, and then somebody on LinkedIn replied to my announcement with the question I'd been quietly avoiding:

> *"Have you ever run your CSP analyzer against your own site?"*

I had not.

## The first run

I pasted the live `Content-Security-Policy` header from [marwandiallo.com](https://marwandiallo.com) into my own analyzer. Three findings came back.

```
1 high · 2 low

CSP02 — 'unsafe-inline' on script-src       (high)
CSP10 — 'unsafe-inline' on style-src        (low)
CSP11 — No CSP reporting configured         (low)
```

I'd built the analyzer carefully enough that it actually told me the truth: my own personal site, the one I'd been linking from this blog about security, was running a CSP with the most common XSS bypass right there in the script directive.

Better that I found it than that someone else did.

## The fixes (three commits, ten minutes)

### CSP02 — `'unsafe-inline'` on script-src (high)

The fix is well understood. Drop `'unsafe-inline'`, add a per-request nonce, and add `'strict-dynamic'`. Next.js 15 already supports this. You set the `x-nonce` request header in middleware and the framework propagates it to its own runtime script tags.

```ts
// middleware.ts
const nonce = Buffer.from(crypto.randomUUID()).toString("base64");
const csp = [
  `script-src 'self' 'nonce-${nonce}' 'strict-dynamic' https://va.vercel-scripts.com`,
  // ...
].join("; ");
```

That single change is the difference between "CSP that exists" and "CSP that meaningfully blocks XSS." `'unsafe-inline'` allows any inline `<script>` block or `onclick=` handler, which is what most XSS payloads actually look like. With nonce + `'strict-dynamic'` in place, a payload can be injected into the DOM and the browser refuses to execute it.

The lab knows this. CSP02 is deliberately written to *not* fire if the policy also has `'strict-dynamic'` and a nonce or hash, because in that combination browsers ignore `'unsafe-inline'` ([CSP3 §6.6.2.4](https://www.w3.org/TR/CSP3/#allow-all-inline)). Most public CSP analyzers I've used don't get that exception right, which was part of what made me want to write this one in the first place.

### CSP10 — `'unsafe-inline'` on style-src (low)

Same fix, applied to `style-src`. Lower severity because CSS-only XSS is much harder, but it isn't zero. Attribute selectors, font ligatures, and scrollbar tricks have all been used to exfiltrate page contents one character at a time. Tailwind compiles to external CSS, so dropping `'unsafe-inline'` on styles cost me nothing.

### CSP11 — no reporting configured (low)

Without `report-to` or `report-uri`, every CSP violation in production happens silently. You don't see the in-the-wild XSS attempts the policy is blocking, and you don't see the well-meaning library you added last week that's now triggering policy failures for real users.

I added a tiny Edge route at `/api/csp-report` that accepts both legacy `application/csp-report` and modern `application/reports+json` payloads, caps body size at 16 KiB, and logs to Vercel runtime logs. Then I added `report-to csp-endpoint; report-uri /api/csp-report` to the policy and the matching `Report-To` response header. One route, one directive, one header.

## Re-test

I pushed the three commits, waited about 90 seconds for Vercel to redeploy, and pasted the new live CSP back into the analyzer.

```
0 high · 0 medium · 0 low · 0 info
```

For independent verification I ran [securityheaders.com](https://securityheaders.com) against the same URL. That passed too.

## Where this kind of tool falls short

A static CSP analyzer is not the same thing as a real-world security audit. It checks a string against a fixed ruleset. Things it does not do:

- Fetch the page and verify the policy is actually enforced (the runtime mode does this now).
- Test whether the nonce is unique per request, which is a critical bug in some homemade implementations.
- Find your *missing* security controls, only your weak existing ones.
- Catch logic flaws, IDOR, broken authZ, or any OWASP Top 10 category that isn't string-shaped.

What it does well is give you a 60-second read on whether the security header you spent half an hour writing is actually doing its job. It's roughly the same category of feedback as ESLint or `cargo clippy`: a fast deterministic linter for one specific class of risk.

The next iteration of the lab takes this one step further. It now has a runtime mode where you paste any URL, the server fetches the live response headers, and the same ruleset runs against the actual policy on the wire. That closes one of the gaps above; you can verify that what you *think* you ship is what you actually ship.

## The lesson I keep relearning

Every consultant has had this conversation. *"You secure other people's stuff for a living. What does your own site look like?"* For most of my career the honest answer has been "I'd rather not show you." Nothing was on fire, exactly. The triage that makes you good at this work in production is the same triage that gets deferred on the personal weekend project.

Building the lab gave me a forcing function. Leaving CSP02 in place while the analyzer that flags it sits at `lab.marwandiallo.com/csp/analyzer` was no longer just an open finding. It was actively embarrassing. The labs got me to fix the site, and I think that pattern generalizes: build the tool that scores the thing, then run it on yourself before anyone else does.

If you want to do the same exercise on your own site, [the CSP analyzer is here](https://lab.marwandiallo.com/csp/analyzer). Paste your header into the first textarea, or drop a URL into the scanner mode and let it fetch your live policy. If you find something interesting, or find a bug in the analyzer itself, I'd love to hear about it.
